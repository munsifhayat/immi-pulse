3. Backend schema extension — current vs extended

The Pydantic QuestionField in schemas.py grows two optional fields: logic (visibility + required-if rules) and flags (outcome tags for Flow Preview colour-coding). Because the column type is JSONB, this is a code-only change — no Alembic migration. A Pydantic root validator enforces that every field_key referenced by a rule exists in the same form.

4. Rule evaluation — client-side live + server-side at submit

The rules engine runs in two places. Client-side (TypeScript port) evaluates rules in real time as the applicant types, showing or hiding fields without a network round-trip. Server-side (Python) re-evaluates at submission to drop answers for fields that were hidden — preventing tampering and validating required-if rules. Both implementations share a JSON fixture of { rules, answers, expected_visible } tuples to stay in lockstep.

5. UI architecture — builder + preview component tree

The builder page wraps a segmented control. Standard reuses today's field editor unchanged. Advanced mounts a two-pane canvas: the field list on the left (each row owns a RuleEditor popover) and the preview pane on the right with two tabs. The QuestionnaireRenderer is extracted into a shared component so the Live Preview and the public fill page render identical output — single source of truth.

6. Data flow — from draft to pre-case

A logic-bearing form follows the same lifecycle as today's flat forms — versioning, public link, submission, downstream pre-case creation, and async AI triage. The only thing that changes is what's inside QuestionnaireVersion.schema: now each field can carry logic and flags. Past responses still snap to their version's frozen schema, so historical data is never affected by edits.

8. Risk matrix

Risk Severity Likelihood Mitigation
Rule references a deleted/renamed field key High Medium Pydantic root validator rejects on save; UI shows inline red border + tooltip on the offending field row; "Rename field" action in the builder offers to auto-update all rule references.
Client-side and server-side rule evaluation drift High Medium Single JSON fixture (tests/fixtures/rule_eval_cases.json) of { rules, answers, expected_visible } tuples — consumed by both BE pytest and FE vitest. CI fails if either implementation drifts.
QuestionnaireVersion table floods on every keystroke save Medium High Only bump a new version on explicit publish; draft state lives in a transient column or localStorage. Confirm with service.py — today every update_questionnaire() with field changes bumps; we'll gate behind a publish=true flag.
React Flow auto-layout looks ugly for forms with many fields/rules Medium Medium Cap recipe complexity at ~15 fields and ~8 rules for v1; use dagre for layered layout; add a manual "fit to view" button. If still ugly at v1, ship Flow Preview behind an "early access" badge.
Past responses break when schema shape evolves (e.g. logic field added) High Low Versioning already solves this — responses snap to the QuestionnaireVersion they were submitted against. Pydantic schema for QuestionField uses logic: Optional[FieldLogic] = None so v1 schemas deserialize fine.
Tampered submission with extra answers for hidden fields Medium Low Server-side rules engine drops answers for hidden fields before persisting; never trust the client to filter. Validation error if a required-visible field is missing.
Public fill page mobile UX degrades with many conditional fields Medium Medium QuestionnaireRenderer is mobile-first; conditional fields use framer-motion for smooth show/hide; explicit mobile test pass before ship (Gideon asked about this specifically on the call).
Engineer drags in @dnd-kit for drag-reorder when not strictly needed Low Medium v1 uses up/down arrows (already implemented); defer drag-reorder to v1.1. Avoids adding a dep just for nicety.
Explicitly cut from v1:
  • Mode 2 — Recommended (AI-drafted forms via Bedrock). Wait until usage/cost shape from existing AI calls settles. Estimated as a separate 1-week sprint once unlocked.
  • Calculated fields (age from DOB, visa expiry countdown). Nice-to-have; the rules engine is forward-compatible.
  • OR-composition of rules. v1 is AND-only between rules on a single field. OR can be layered on without breaking existing forms.
  • Editing logic from the Flow Preview canvas. v1 is read-only; edits happen via RuleEditor popovers in the field list.
  • Drag-and-drop field reordering. Up/down arrows ship; @dnd-kit waits for v1.1.

Key takeaways
  • Zero migrations. The JSONB schema column means we ship the entire logic layer as a code-only change.
  • Single source of truth for rendering. Extracting QuestionnaireRenderer means the consultant's Live Preview and the applicant's actual form are guaranteed identical — no "looked fine in preview" surprises.
  • Two-place rules engine, one fixture. TS and Python implementations share a JSON test fixture so CI catches drift before it ships.
  • P4 is the bottleneck. Logic editor (Phase 4) blocks Live Preview, Flow Preview, and the public-page logic — front-load it.
  • Phase 1 is shippable on its own. Better Standard templates can ship to Gideon on day 2 as an early signal, even before Advanced lands.
  • React Flow is already paid for. @xyflow/react@^12.10.1 is in package.json — no new dep, no bundle-size justification needed.
