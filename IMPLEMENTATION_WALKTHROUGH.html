What the applicant sees

  • Branded form with consultant's firm name + OMARA #
  • "Thanks, we'll be in touch" confirmation on submit
  • No account creation — friction-free intake

What happens server-side

  • POST /public/q/{slug}/submit (rate-limited by IP-hash)
  • Creates QuestionnaireResponse (answers JSONB, ip_hash, ua)
  • Finds-or-creates Client + ClientOrgLink
  • Creates PreCase (status=pending, ai_status=pending)
  • Enqueues async triage() job
6
AI gateway · Bedrock

Background — AI triages the lead

Two Bedrock models run in sequence. Haiku 4.5 extracts structured fields from the prose answer; Sonnet 4.5 produces a fit assessment with a confidence score. Cost and tokens are deducted from the org's CreditWallet.

INPUT
Questionnaire response (JSONB)
5 fields, ~180 words of free text
→
HAIKU 4.5 · classifier
Extract: visa subclass, exp years, country, sponsor
~340 in / 120 out tokens
→
SONNET 4.5 · analyzer
Suggested outcome + confidence
~520 in / 280 out tokens
PRECASE.AI_SUMMARY (output)
{
  "outcome": "likely_fit",
  "confidence": 0.87,
  "summary": "Senior SWE, UK passport, 7y exp, has confirmed AU sponsor.
   Strong fit for 482 TSS Medium-Term stream. Standard documentation
   pathway. Prioritise English evidence + skills assessment review.",
  "next_steps": ["confirm sponsor SBS approval", "request CV + IELTS", "book 30-min consult"]
}

Outcome buckets

  • likely_fit — ready to convert to paying case
  • needs_info — request more docs before quoting
  • paid_consult — borderline, charge for assessment
  • unlikely_fit — politely decline / refer out

What happens server-side

  • AIGateway.classify() → Haiku via Strands SDK + Bedrock
  • AIGateway.analyze() → Sonnet, system prompt seeded with org niche
  • Persists ai_summary, ai_suggested_outcome, ai_confidence on PreCase
  • Logs to AIUsageLog, debits CreditWallet

Checklist tab (separate)

  • Generate from visa template — Sonnet drafts a tailored checklist for the visa subclass
  • 12-item checklist for 482 (passport, sponsor, IELTS, skills assessment, etc.)
  • Each item links to a document; status pending → uploaded → validated
  • Client sees the same list in their portal

What happens server-side

  • POST /cases/{id}/documents → S3 upload (presigned), CaseDocument row created
  • Async DocumentAnalyzer agent → extracts text, classifies type, flags issues
  • POST /cases/{id}/generate-checklist → Sonnet generates checklist into case.metadata.checklist
  • GET /cases/{id}/documents/{doc_id}/download → 5-min presigned URL

Why this design

  • No password creation — friction for clients is brutal
  • Token in URL + PIN out-of-band = two factors
  • Branded with the firm's name (whitelabel feel)
  • 15-minute session, then re-PIN

What happens server-side

  • POST /cases/{id}/generate-portal-link → CasePortalToken (token + bcrypt-hashed PIN, 7d expiry)
  • POST /public/portal/verify → validates token+PIN, issues short-lived session JWT
  • Portal-uploaded docs land in same case with uploaded_by_type=client_portal

Most consultants charge in stages. Checkpoints are simple: amount + description + due date. Send to client (currently a logged-not-emailed action), mark paid, and the case timeline records it.
